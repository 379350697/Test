# QuantX 代码审查与测试结果（更新版）

## 1. 单元测试与覆盖率要求检查

### 1.1 当前是否有单元测试

有。当前仓库存在 `tests/test_quantx.py`，包含 4 个测试函数，覆盖了以下主链路：

- 内置策略注册数量校验
- 回测与 walk-forward
- 自定义策略仓加载 + 报告可复现字段
- tick/orderbook 回测、执行器、监控、A/B、在线学习入口

### 1.2 覆盖率要求是否已定义

当前 **未定义覆盖率门禁**：`pyproject.toml` 未包含 `pytest-cov`、`coverage` 配置或最小阈值要求（例如 fail-under）。

### 1.3 覆盖率现状

当前环境缺少覆盖率工具链（`pytest-cov`/`coverage`），无法输出精确覆盖率百分比；建议在 CI 固化：

- `pytest --cov=quantx --cov-report=term-missing --cov-fail-under=80`

---

## 2. 边界情况处理检查

### 2.1 已有处理（正向）

- 监控模块对空权益曲线直接返回空告警与 0 回撤，避免异常。
- 执行模块限制 mode 仅 `paper/live`，并对 `disabled_or_killed` 拒单。
- 回测模块对关键风险路径有硬约束：最大回撤触发平仓、冷却期、日内最大下单数、ATR/Donchian/时间止损等。
- `twap/vwap` 下单的分片数量做了下界保护（至少 1 片）。

### 2.2 仍需增强（风险点）

- CLI 中多处 `json.loads(args.xxx)` 无统一输入校验，非法 JSON 会中断命令。
- 执行层缺乏统一参数合法性校验（如负数 `qty`、非 BUY/SELL 输入）。
- 回测参数校验依赖调用方约定，缺统一 schema（可考虑 pydantic/dataclass validator）。

---

## 3. 性能与安全约束检查

### 3.1 性能

- 已支持并行回测（`ProcessPoolExecutor`）。
- 但缺乏可量化性能门槛（耗时、吞吐、内存峰值）及回归基线测试。

### 3.2 安全/风控

- 风险参数有默认上限结构：`max_position_pct`、`max_drawdown_pct`、`cooldown_bars`、`max_orders_per_day`。
- 执行层有 `arm`、`kill_switch`、`close_all` 等安全控制。
- 建议补充命令级审计字段（cmd/symbol/strategy/error_code/latency）统一化。

---

## 4. 错误处理与日志检查

### 4.1 现状

- `data-fetch-klines` 有 `try/except` 并输出结构化错误。
- 报告渲染对 `matplotlib` 缺失有降级兜底。
- 监控日志分析提供 error/kill_switch/order/reject 计数聚合。

### 4.2 缺口

- 全局缺少统一错误码体系和异常分层（输入错误、业务约束错误、基础设施错误）。
- 目前日志为字符串拼接，未完全结构化，追踪与审计可读性有提升空间。

## 7. 下一步：明确细化需求 + 强制质量标准 + 分步计划

> 原则：不一口气大改，按小步迭代，每步都“改动-审查-测试-复盘”。

### Step 1：需求上下文模板（必须先补齐）

- 业务目标：研究/模拟盘/实盘前验证？
- 范围边界：只改 CLI？还是回测引擎+执行层一起？
- 兼容性：CLI 参数与输出格式是否必须完全兼容？
- 风险等级：是否允许行为变更，是否需要可回滚开关？

### Step 2：质量门禁（DoD）

- Lint：`ruff check .` 必须通过。
- 测试：`pytest -q` 必须全绿。
- 覆盖率：`--cov-fail-under=<阈值>`（建议 80 起步）。
- 审查：每步必须给出“风险点 + 回滚点 + 测试摘要”。

### Step 3：迭代路线（每次一个小 patch）

- 迭代 A：输入校验与错误码框架（不改业务逻辑）。
- 迭代 B：补边界测试与异常路径测试。
- 迭代 C：性能 smoke + 指标基线。
- 迭代 D：日志结构化 + 审计字段统一。

### Step 4：每轮交付输出

- 改动摘要（哪些文件、为什么）
- 代码审查结论（P0/P1/P2）
- 测试摘要（命令 + 结果）
- 下一轮建议（只给最小可行范围）
