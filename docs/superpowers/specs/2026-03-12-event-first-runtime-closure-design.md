# Event-First Runtime Closure Design

## Summary

This spec refines the existing unified derivatives runtime design for a personal trading system.

The new priority is:

- `event_strategy` first, because the primary live path is `scalping / orderbook / tick-driven`
- `bar_strategy` second, but on the same runtime contract
- one execution truth across `backtest`, `paper`, and `live`
- paper trading and backtesting should be realistic enough to calibrate against live behavior

This document does not replace the shared-runtime direction in the earlier design. It tightens the target around the parts that matter most for a personal system:

- strategy closure
- runtime closure
- replay closure
- live calibration closure

The core principle remains:

`strategies may differ, execution semantics may not`

## Why This Follow-Up Exists

The current implementation has important pieces of the runtime foundation, but it does not yet satisfy the design-level closure required for personal production use.

Current strengths:

- normalized runtime events exist
- shared order, ledger, fill, and adapter modules exist
- paper execution uses the runtime order and ledger path
- OKX and Binance adapters map exchange payloads into a shared schema

Current gaps:

- strategy code still centers on `signal(...) -> int`, not `OrderIntent`
- `event_strategy` and `strategy_runtime` do not yet exist as first-class interfaces
- `backtest` still mutates legacy cash/position state first and only mirrors into runtime traces afterward
- `live` still treats the runtime mostly as normalized event output instead of the single execution truth
- replay drift metrics are not yet generated by re-running a paper model against live market events

Because this is a personal system, correctness of the loop matters more than breadth of features. The first milestone should therefore optimize for a hard, auditable execution loop instead of interface completeness on paper.

## Product Scope

### In Scope

- one personal trading account
- one active exchange rollout path at a time
- `USDT`-margined perpetuals
- `cross margin`
- `hedge mode`
- `event_strategy` as the first fully closed strategy path
- `bar_strategy` on the same execution contract
- realistic `paper` and `backtest` paths
- drift calibration from live captures

### Out of Scope For The First Closure

- multi-account orchestration
- spot trading
- options
- multiple live execution instances for one account
- full exchange-order-type coverage
- exact exchange-matching parity in every microstructure edge case

## Design Goals

- Make `event_strategy` the primary production path for scalping and orderbook-driven logic.
- Keep `bar_strategy` as a first-class interface for CTA and trend systems without allowing a second execution semantics.
- Ensure `backtest`, `paper`, and `live` differ only by data and fill sources, not by state-machine rules or ledger rules.
- Make replay the basis for validation, daily review, and parameter calibration.
- Keep the first production rollout narrow enough for a personal operator to understand and supervise.

## Strategy Model

The system should expose two strategy interfaces.

### 1. `event_strategy`

Primary use cases:

- scalping
- orderbook imbalance
- trade impulse detection
- tick-by-tick execution logic
- microstructure-sensitive confirmation

Interface:

`on_event(ctx, event) -> list[OrderIntent]`

### 2. `bar_strategy`

Primary use cases:

- CTA
- trend following
- slower breakout systems
- lower-frequency confirmation logic

Interface:

`on_bar(ctx, bar) -> list[OrderIntent]`

### Shared Contract

Both interfaces must produce the same output type:

- `OrderIntent`

Strategies must never:

- mutate positions
- mutate balances
- mark orders as filled
- bypass risk checks
- talk to exchange clients directly

This keeps strategy variation at the decision layer only.

## Strategy Runtime

Add a dedicated `strategy_runtime` layer between incoming events and the shared execution core.

Responsibilities:

- route normalized inputs to the appropriate strategy interface
- maintain strategy-local state only
- expose read-only runtime views through `ctx`
- emit `OrderIntent` objects
- stamp each intent with tracing metadata for replay and attribution

Recommended intent metadata additions:

- `intent_id`
- `strategy_id`
- `signal_id`
- `reason`
- `created_ts`
- `tags`

These fields are needed for an auditable strategy-to-order-to-ledger trail.

## Unified Execution Loop

All runtime modes should follow the same loop:

1. receive normalized event
2. feed `strategy_runtime`
3. collect `OrderIntent`
4. run runtime risk validation
5. create or amend tracked orders through `order_engine`
6. receive downstream order/fill/account events from the active fill source
7. apply events to `ledger_engine`
8. append all normalized events to replay storage
9. emit monitoring and drift-analysis artifacts on the cold path

This loop is the single truth path.

### Runtime Modes

The runtime mode may change:

- market data source
- fill source
- account event source

The runtime mode must not change:

- strategy contract
- order state machine
- ledger semantics
- risk validation
- funding and fee booking semantics
- replay event model

## Realistic Backtest And Paper Design

### Primary Backtest Path: `event_backtest`

This is the preferred backtest path for `event_strategy`.

Inputs:

- historical trades
- historical orderbook updates
- mark price updates
- funding events
- normalized order/account events when available

Behavior:

- strategy reads normalized historical events
- strategy emits `OrderIntent`
- the same runtime core processes order state and ledger updates
- simulated fills are generated by the same fill-model family used by paper mode

This is the only backtest path suitable for live drift calibration.

### Secondary Backtest Path: `bar_backtest`

This path is allowed only as a lower-fidelity compatibility layer.

Inputs:

- candles or bars

Behavior:

- `bar_strategy` emits `OrderIntent`
- runtime core remains the execution truth
- reports must explicitly label this path as lower fidelity

Recommended report field:

- `fidelity = low`

### Paper Mode

`paper` mode should be defined as:

`live market data + local exchange simulator + shared runtime ledger`

Paper mode should not directly mutate positions or balances. It should produce synthetic exchange-like events that flow through the same order and ledger path as live.

Minimum realism requirements for first closure:

- queue delay
- partial fills
- cancel delay
- side-aware slippage
- exchange-style reject paths
- stale-market-data handling
- fees
- funding
- cross-margin accounting

## Drift Calibration Loop

The system should treat drift measurement as a first-class daily process.

### Daily Flow

For each live trading day:

1. capture normalized `market`, `order`, `fill`, and `account` events from live
2. rebuild the live ledger from the captured tape
3. re-run `paper` on the same market-event tape
4. compare the two outcomes
5. recalibrate fill-model parameters if drift exceeds threshold

### Required Metrics

- `order_state_match_rate`
- `fill_price_drift`
- `partial_fill_ratio_drift`
- `cancel_success_drift`
- `equity_drift`
- `available_margin_drift`
- `funding_booking_drift`

### Important Constraint

`paper` drift must not be computed by summarizing the same live order/fill results twice.

Instead:

- live summary must come from captured live execution events
- paper summary must come from re-running the paper simulator against the captured market tape

Without this separation, drift reports become tautological and cannot calibrate anything.

## Personal-System Rollout Plan

Because this system is personal, the first rollout should be deliberately narrow.

### Stage 1: Replay + Paper Closure

Target:

- `historical replay`
- `paper`
- daily drift report

Required gate:

- one strategy contract for event-first runtime
- deterministic replay reconstruction
- paper/live comparison artifacts produced daily
- no unexplained state reconstruction gaps

### Stage 2: `OKX` Micro-Live

Target:

- smallest notional possible
- strict symbol whitelist
- one strategy instance

Required gate:

- stable Stage 1 drift behavior
- restart takeover report passes
- operator can explain rejects, cancels, fills, and ledger movements

### Stage 3: `OKX` Normal Live

Target:

- routine live operation on the primary personal account

Required gate:

- drift remains inside configured thresholds over time
- funding, fee, and margin numbers reconcile consistently
- repeated restart and recovery exercises succeed without unexplained differences

### `Binance`

`Binance` should remain a second exchange on the same core, not a second rollout truth.

The production reference path remains:

- `OKX` first

## Testing Strategy

Testing should be organized around closure, not only local correctness.

### 1. Unit Tests

Prove correctness of:

- order state transitions
- hedge-mode ledger math
- cross-margin risk checks
- fill simulation primitives

### 2. Parity Tests

Prove that the same intent family yields the same state progression across:

- `event_backtest`
- `paper`
- `live_replay`

### 3. Golden Replay Tests

Use captured `OKX` event tapes as replay fixtures.

These tests should verify:

- order-state sequence reconstruction
- ledger reconstruction
- funding and fee replay
- restart takeover reconstruction

### 4. Drift Acceptance Tests

Per strategy and symbol family, assert maximum tolerated drift across the required metrics.

### 5. Operational Failure Tests

Cover:

- disconnects
- duplicate event delivery
- out-of-order events
- missing private-stream gaps
- restart recovery
- safe downgrade to paper or read-only modes

## Migration Guidance

The next implementation pass should focus on closure, not breadth.

Recommended order:

1. introduce `strategy_runtime` and convert strategies to emit `OrderIntent`
2. make `event_strategy` the first fully closed path
3. convert `backtest` from legacy cash/position mutation to runtime-owned execution
4. make `paper` a proper exchange simulator backed by the shared runtime
5. rework replay so paper-vs-live drift uses separate execution paths on the same market tape
6. move live execution from "normalized output" to "runtime-owned state"
7. reattach `bar_strategy` as a compatibility layer on the same runtime

## Acceptance Criteria

This design is accepted when the system can demonstrate:

- one strategy contract family: `event_strategy` and `bar_strategy`, both outputting `OrderIntent`
- one shared execution semantics across `backtest`, `paper`, and `live`
- no direct position or balance mutation from strategy code
- `event_strategy` as the first production-closed path
- realistic `paper` behavior with queue, partial-fill, cancel-delay, and slippage modeling
- drift reports that compare live execution against a separately re-run paper path
- staged `OKX` rollout from replay and paper, to micro-live, to normal live
